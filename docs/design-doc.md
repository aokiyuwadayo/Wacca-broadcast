# YUWA Broadcast 設計ドキュメント（v0.1）

> **作成日**: 2026-05-29
> **対象読者**: 青木（PO）、相談する起業部の運営メンバー、将来の実装担当
> **目的**: 「何を・なぜ・どう作るか」を1枚で共有する。実装の地図であり、運営メンバーへの説明の起点。
> **状態**: 設計確定 ＋ 実地検証済（§13）。実装は Phase A から。

---

## 1. これは何か

サークルの告知（毎週の活動・単発イベント）を、**ふんわりメモから AI が整え、各プラットフォーム向けに書き分けて、配信（or コピペ）まで地続きにする**ツール。投稿忘れ防止のリマインドも持つ。

- **まずは** 青木（起業部 部長）専用の個人ツールとして作る。
- **将来は** 他サークルの運営にも使ってもらえるよう、設定・テンプレ・配信先は最初から「アカウント（サークル）単位」で持つ（マルチテナント前提）。
- 技術構成は姉妹プロジェクト [YUWA](https://github.com/aokiyuwadayo/yuwa) と揃え、後で「YUWA の一機能」として吸収できる形にしておく。

### なぜ独立ツールとして始めるか
- YUWA 本体はボランティア部員エンジニアが Phase 1→2 を実装中。**計画外の重い機能（外部連携・スケジューラ）を混ぜてエンジニアの負担を増やさない。**
- このツールは**部員の個人情報を一切扱わない**ので、YUWA 本体の個人情報・規約リスクと完全に切り離せる。

---

## 2. 解決する痛み

| # | 痛み | 効かせ方 |
|---|---|---|
| 1 | 投稿先が多い（二重投稿） | 1入力 → 各PFへ一斉配信 / コピペ |
| 2 | 見やすい文章を作るのが面倒 | 統一テンプレ＋AIが各PF最適化で生成 |
| 3 | 先延ばし・投稿忘れ | 定期下書きの自動生成＋Slackリマインド |

インパクト: 1回 30分〜1時間 × 毎週 ＝ 年間 25〜50時間。

---

## 3. コンセプト・全体フロー

```
[ふんわりメモを入力]
      ↓
[AIと相談して内容を固める]   ← ガイド型（1問ずつ）⇄ フォーム型 を切替可
      ↓
[中間JSON に整理]            ← すべての心臓（§4）
      ↓
[各PF向けに自動で書き分け生成]  ← LINE / Teams / (将来)Discord
      ↓               ↘
[配信 or コピペ]        [スライド下書き生成（将来 Phase F）]
   ├ Teams   : 自動 / 予約
   ├ Discord : 自動（将来）
   └ LINE    : コピペ（無料枠の都合で当面これが最適）
      ↓
[予約投稿 & リマインド]       ← Slack に通知。投稿忘れ防止
```

## 4. 中間JSON スキーマ（すべての心臓）

入力を一度この構造に正規化し、ここから**各PF文面もスライドも派生**させる。修正は「この JSON を直す → 全PF再生成」で一貫性を保つ。

```jsonc
{
  "kind": "activity" | "event",       // 活動（内部連絡）か イベント（集客）か
  "title": "立花祭の出展内容を決める回",
  "datetime": {
    "start": "2026-06-04T16:30",
    "end": null                         // 未定なら null
  },
  "location": { "name": "E棟3階 R教室", "access": "", "online_url": "" },
  "summary": "立花祭（福岡工業大学の文化祭）の出展内容を決める",  // 今回やること（必須）
  "body": ["活動スライドで方向性共有", "出展アイデア出し", "出展内容を決定"], // 流れ。AIが提案→ユーザー確認で確定（勝手に作らない）
  "bring": [],                          // 持ち物・事前準備
  "rsvp": { "method": "", "deadline": "" }, // 参加・欠席連絡
  "fee": null,                          // イベント時の参加費
  "guests": [],                         // イベント時の登壇者
  "cta": null,                          // イベント時の拡散依頼
  "hook": "おもしろい立花祭にできるかは今回次第！", // 今回の推しポイント／意欲喚起
  "note": "",                           // 補足メモ（会場固有の事情等はここで吸収）
  "links": [{ "label": "", "url": "" }],
  "images": []                          // 添付画像（§7）
}
```

- **必須**: `title` / `datetime.start` / `location.name` / `summary`
- 残りは任意。無ければ各PF文面で省略する。
- 会場固有の事情（例: 「19時に教室が閉まる」）は専用欄を作らず `note` に入れる ＝ 一般化しすぎない。

## 5. テンプレート（2種）

### 活動告知（毎週の連絡。8項目・長ければ短縮）
タイトル / 日時 / 場所 / 今回やること / 内容・流れ / 持ち物・準備 / 参加・欠席連絡 / 部長から一言

### イベント告知（単発の集客。来たくなる・拡散したくなるトーン）
イベント名 / 日時 / 場所・アクセス / 概要・見どころ / こんな人におすすめ / 参加方法・申込 / 参加費 / ゲスト・登壇者 / 拡散のお願い

## 6. プラットフォーム別 書き分けルール

| | LINE版 | Teams版 | Discord版（将来） |
|---|---|---|---|
| 長さ | 300〜500字・1メッセージ完結 | 制限ゆるめ・詳しめOK | 中くらい |
| 体裁 | 絵文字を見出しに・空行多め | 見出し＋箇条書きで整理 | Markdown（**太字**・見出し） |
| 絵文字 | 多め | 控えめ | 多め |
| トーン | フランク・親しみ | 整然・情報重視 | カジュアル |
| リンク | 末尾にまとめる | 本文中に可 | 本文中に可 |

## 7. 入力UX

- **ガイド型 ⇄ フォーム型 を切替可能**（その時の気分・状況で選ぶ）。
  - ガイド型: 「全部ダンプ → AIが足りない必須項目だけ1問ずつ質問」。頭の中にある人は速く、ない人はガイドされる。
  - フォーム型: テンプレ項目が並ぶフォームを直接埋める。
- **生成後はチャットで微調整**（「もっと短く」「この項目消して」「もっと熱く」）→ 中間JSONを直して全PF再生成。
- **画像の挿入**: 相談・フォームの最中に画像を足せる。Discord/Teams は埋め込み、LINE はコピペ＋画像添付。

## 8. オンボーディング方針（他サークルにも使ってもらう本丸）

**鉄則: コピペは全PF day-1の標準装備。自動投稿は任意のアップグレード。** 技術力ゼロでも今日から使えて、欲しい人だけ自動化に進める。

| PF | day-1（設定不要） | アップグレード |
|---|---|---|
| LINE | ✅ コピペ | （当面なし。無料枠が厳しくコピペが最適） |
| Discord | ✅ コピペ | Webhook URL を貼る（5クリック・コード不要） |
| Teams | ✅ コピペ | Power Automate Workflow を画像手順で（中難度） |
| Notion | ✅ コピペ | トークンを貼る |

- 接続ウィザードは**画像/GIF付きステップ＋「テスト送信」ボタン**。実際に1通届くまで“接続済み”にしない（silent fail 防止）。

## 9. 神機能（ただの整形ツールを超える差別化）

1. **定期告知の自動下書き＋Slack促し** ← 先延ばし対策の決定版。「毎週木曜18時」を登録 → 活動2日前に半分埋まった下書きが自動生成され Slack に「確認して出して」。
2. **前回複製** — 先週の告知を複製→日付だけ変えて即完成。
3. **文体学習** — 過去の手直し最終版を学習し「自分っぽい初稿」を出す（few-shot で実装）。
4. **履歴・検索・再利用** — 全告知が残る。
5. **ワンタップ全PF配信** — 確認したら接続先へ一斉／予約。

汎用AIチャットとの差＝「型がある × 各PF最適化 × 配信まで地続き × 忘れない × 自分の文体」の5点セット。

## 10. 権限・引き継ぎ

- `account`（サークル）に複数 admin。設定・テンプレ・配信先は**人ではなくサークルに紐づく**ので、卒業で人が変わっても資産が残る。owner権限の移譲だけで引き継ぎ完了。
- v1 は青木1人（owner）。器だけ先に用意。

## 11. 失敗時の挙動

- 配信はPFごとに独立ログ。1つ失敗しても他は成功。
- 失敗 → 自動リトライ（数回）→ ダメなら Slack に「○○投稿失敗、コピペはこれ」＋手動再送ボタン。
- “成功”を騙らない：テスト送信も配信も**実APIレスポンスで確認**。

## 12. 技術構成とコスト

| 領域 | 技術 | 備考 |
|---|---|---|
| フロント | Next.js + Tailwind + shadcn/ui | スマホ対応PWA。YUWAと同じ |
| DB/認証/秘密/cron | Supabase | pg_cron で予約・リマインド。Vault でトークン暗号化 |
| 生成・修正 | Claude API | Haiku 主力 |
| ホスティング | Vercel | — |

**コスト（2026-05 裏取り）**
- v1（青木1人・非商用）は**ほぼ無料**。Claude Haiku ≈ 1本9円（月数百円）＋ .club ドメイン年約2,000円。Supabase Free（pg_cron 込み）＋ Vercel Hobby で土台ゼロ円。
- **有料化の分岐点**: ①**他サークル開放＝商用化 → Vercel Pro $20/月が即必須**（寄付すら商用扱い）②Supabase Pro $25/月（容量増・Free一時停止回避）③配信量・モデル増。

**実現性メモ**
- Teams: 旧 Incoming Webhook は 2026-05 廃止 → **Power Automate Workflows** が現行。
- LINE: 日本は月200通無料・**送信先人数分カウント**で超過しやすい → **当面コピペ固定が正解**。LINE Notify は 2025-03 終了で選択肢外。
- Discord: Webhook で無料・簡単。
- スライド: Adobe Express は裏での全自動生成が弱い → **python-pptx / Google Slides API のテンプレ差し込み**に寄せる。

## 14. 実地検証ログ（2026-05-29）

コードを書く前に、Claude を「Phase A のエンジン」に見立てて本番データで1本実演した。

- **入力**（ふんわりメモ）: 「来週は立花祭という福岡工業大学の文化祭の出展内容について決める。16:30〜 e棟3階R教室」
- **ガイド型で穴を質問** → 木曜（6/4）/ 終了は19時に教室クローズ / 持ち物なし / スライドで意欲喚起、を回収。
- **中間JSON に整理 → LINE版・Teams版を生成**（§4 のスキーマはほぼそのまま機能）。
- **得られた告知文は今週そのままコピペで使える実物になった。**

### 検証で出た学び（設計に反映済）
1. ガイド型だけでなく**フォーム型との切替**が欲しい → §7 に反映。
2. **画像挿入**機能が欲しい → §7 に反映。
3. 会場固有の事情（教室クローズ時刻）は**専用欄にせず `note` で吸収** → §4 に反映（over-engineering回避）。
4. 流れ（body）は**AIが提案→ユーザー確認**で確定（勝手に作らない）→ §4 に反映。
